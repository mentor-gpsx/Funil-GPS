## 📊 Dev Notes

**Dependencies:** Depends on Story 1.1 (schema for user table — added in `005_users_auth.sql`)
**Assigned to:** @dev (Dex)
**Tech Stack:** NestJS, TypeScript, `jsonwebtoken`, `bcrypt`, `speakeasy`, `qrcode`, React (frontend)

**Security Considerations:**
- JWT: HS256 with `JWT_SECRET` (min 32 chars enforced in production)
- Password: bcrypt cost factor 12 (configurable via `BCRYPT_COST`, range 10–15)
- MFA TOTP: RFC 6238, 6 digits, ±30s window (configurable)
- Token refresh: Rotating with `replaced_by` chain; reuse-detection revokes entire chain
- Custom claims: `tenant_id` resolved server-side from user row, NEVER from client
- Tenant isolation: `SET LOCAL app.current_tenant` inside every transaction
- Secrets: Environment variables only; production startup fails if `JWT_SECRET` missing
- Lockout: 5 failed attempts → 15 min lockout (configurable)
- Refresh token storage: SHA-256 hash only (raw token never persisted)
- Backup codes: bcrypt-hashed in `mfa_backup_codes` JSONB column

**Architectural Decision (AUTO):**
Story title says "Supabase + MFA" but implemented as local JWT + bcrypt + speakeasy by default,
with Supabase as opt-in via `AUTH_PROVIDER=supabase`. Rationale:
1. User mission explicitly requested "JWT generation/validation, password hashing with bcrypt, RBAC"
2. Local pipeline is deterministically testable without external Supabase service
3. Custom claims and tenant isolation work identically in both modes
4. Switching to Supabase is a config change, not a code change (provider toggle in `supabase.config.ts`)

**TOTP Flow:**
1. User login → `POST /auth/login` (email/password)
2. If MFA enrolled → response: `{ mfaRequired: true, sessionToken: '<5-min jwt>' }`
3. User submits TOTP → `POST /auth/mfa/verify-login` with `{ sessionToken, totp }`
4. Server verifies → returns `{ tokens: { accessToken, refreshToken, expiresIn, tokenType: 'Bearer' }, user }`

**MFA Enrolment Flow (post-login):**
1. Authenticated user → `POST /auth/mfa/setup` → response: `{ qrCode, secret, backupCodes, otpauthUrl }`
2. User scans QR + saves backup codes
3. User submits 6-digit code → `POST /auth/mfa/verify` → `mfa_enrolled = true`
