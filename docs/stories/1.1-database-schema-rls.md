## ✅ Acceptance Criteria

- [ ] PostgreSQL schema deployed with 3 core tables:
  - [ ] `chart_of_accounts` (tenant_id, id, code, name, account_type, parent_id, is_active)
  - [ ] `journal_entries` (tenant_id, id, entry_date, description, posted_at, posted_by, is_reversed, reversal_of)
  - [ ] `journal_entry_lines` (tenant_id, id, entry_id, account_id, debit, credit, line_order)

- [ ] RLS policies enforced:
  - [ ] `tenant_isolation` policy on all 3 tables
  - [ ] `SET LOCAL app.current_tenant` middleware enforcement tested

- [ ] 5 critical indices created:
  - [ ] `(tenant_id, entry_date DESC)` on journal_entries
  - [ ] `(tenant_id, account_id, entry_id)` on journal_entry_lines
  - [ ] `(tenant_id, code)` UNIQUE on chart_of_accounts
  - [ ] `(tenant_id, posted_at)` partial on journal_entries WHERE posted_at IS NOT NULL
  - [ ] `(tenant_id, entry_id)` on journal_entry_lines

- [ ] Database triggers deployed:
  - [ ] `enforce_balanced_entry()` - validates SUM(debit) = SUM(credit)
  - [ ] `block_posted_mutation()` - prevents editing posted entries
  - [ ] `one_sided_lines()` - ensures debit XOR credit (not both)

- [ ] Partitioning configured:
  - [ ] RANGE partitioning on journal_entry_lines by entry_date (monthly/annual)
  - [ ] tenant_id as leading key in all local indices

- [ ] Test data created:
  - [ ] 2 test tenants for cross-tenant validation
  - [ ] Sample chart of accounts per tenant
  - [ ] Sample journal entries (posted and draft)

- [ ] Cross-tenant isolation validated:
  - [ ] SELECT isolation: Tenant A cannot see Tenant B data
  - [ ] INSERT isolation: Attempting cross-tenant insert fails
  - [ ] UPDATE isolation: Cannot update records from other tenant
  - [ ] Trigger validation: Balance enforcement works per entry

## 🔐 Multi-Tenancy Validation

**Test Scenario 1:** SELECT isolation
```sql
-- As tenant_A, should see only own data
SET app.current_tenant = 'tenant-a-uuid';
SELECT * FROM journal_entries; -- Only tenant A entries
```

**Test Scenario 2:** Insert isolation
```sql
-- Attempt to insert for different tenant should fail
SET app.current_tenant = 'tenant-a-uuid';
INSERT INTO journal_entries (tenant_id, ...) VALUES ('tenant-b-uuid', ...);
-- Should fail due to RLS policy
```

**Test Scenario 3:** Trigger enforcement
```sql
-- Imbalanced entry should be rejected
INSERT INTO journal_entries ... (post to journal_entry_lines with unequal debits/credits)
-- Should fail due to enforce_balanced_entry() trigger
```
